# ROTH.C modding SDK

This directory is the machine-readable substrate the modding platform is built on: a
permanent **function-identity table**, a mechanically **extracted signature schema**, the
**generators** that turn those into a typed public header + host adapters + a machine ABI
descriptor, and the **CI gates** that keep the ABI stable across releases.

The design is **schema-first**: hand-maintained JSON in `sdk/schema/` and `sdk/abi/` is the
source of truth, and every author-facing artifact (`sdk/include/roth_sdk.h`,
`sdk/abi/descriptor.json`, `sdk/generated/host_adapters.c`) is **generated** from it and checked
in. CI regenerates in memory and diffs, so the generated files can never silently drift from the
schema.

> The engine stays pristine. The generators read `roth_c/src/engine/engine.h` and the corpus
> metadata; they never edit `roth_c/src/`.

**Writing a mod?** Start with [`MODDING_GUIDE.md`](MODDING_GUIDE.md), which is the author-facing
guide (mod types, contexts, overrides, chains, build recipe). This README covers the schema/CI
machinery underneath it.

## Function identity: the permanent numeric ID

Every engine function has a **permanent numeric ID equal to its address in the original 1996
`ROTH.EXE`**. That address is a historical fact and a frozen ABI forever. The names themselves
are **NOT** the identifier, since understanding of the original functions may change (even slightly)
and functions may need to be renamed.

`schema/function_ids.json` is the identity registry — one row per function:

```json
{
  "id": 118492,
  "va": "0x1cedc",
  "canonical_name": "give_item",
  "aliases": [],
  "subsystem": "inventory",
  "klass": "engine"
}
```

`canonical_name` is the **current linkable symbol** (the `engine.h` name a mod's `ROTH_FN_<name>`
enumerator is derived from). Every row ships with an empty `aliases` list, because the identity
table's names are kept in sync with `engine.h`. When a function is later renamed, the old name is
**appended to that row's `aliases`** so lookups by the old name still resolve to the same ID — the
list only ever grows, so a mod written against any earlier name keeps resolving.

**Maintenance rules (CI-enforced):**

- IDs are permanent and should never be reused, renumbered, and most importantly removed.
- A rename **appends** the old name to `aliases`; aliases are never removed.
- Every corpus address has exactly one row; every name (canonical or alias) resolves to exactly
  one ID.
- `canonical_name` must match the current source symbol. After renaming in `engine.h`, re-run
  `make -C sdk seed-ids` — the maintenance step folds the old name into `aliases` — and review the
  diff before committing.

IDs with no unique 1996 address (multi-entry shared code, aliased-without-address helpers) would
draw a **synthetic ID** from the reserved space at/above `0x80000000`. The current table needs
**zero** of these — all 1471 rows have a unique canonical address.

## The exposed engine surface

24 functions carry `exposed: true`: the subset callable directly through the host API table
(`struct roth_engine_api_v1`), reached as `api->engine->give_item(...)` or via the
`ROTH_CALL_give_item(api, ...)` convenience macro. Each is a proven reuse point used by a shipped
example plugin — item grant + icons, UI text, reloc/DAS image blits, framebuffer
save/clear/dirty-rect, record/menu helpers, pool free, and DOS file open/close.

```c
#include "roth_sdk.h"
uint32_t h = api->engine->give_item(entry_index, 0);   /* or ROTH_CALL_give_item(api, entry_index, 0) */
```

Plugins **only** call engine functions through this table — never by linking an engine symbol
directly. (The full ~1132-function overrideable set is a separate surface; see
_The override-target manifest_ below.)

### THE EXPOSURE-ORDER LAW

The engine-call table's **slot order is the exposure order, and it is a frozen ABI**. The reason
is concrete: the table is a C struct of function pointers, so each slot's **byte offset** is part
of the compiled ABI. A plugin built against one release indexes those offsets; if a later release
reordered or inserted a slot, every offset past the change would shift and the old plugin would
call the wrong function.

So the rule is **append-only, forever**: new exposures are added **only at the end** of the table,
existing slot offsets never move, a published signature is never changed, and a function is never
un-exposed or reordered. Every exposure is therefore a true append-only minor bump. The
`exposure-order` CI gate enforces this by checking that the released baseline's slot sequence is an
exact **prefix** of the current table — a re-sort, insertion, or removal fails hard, and no
version bump excuses it.

## `game_ram`

`game_ram` is the contiguous engine global-state image at the original 1996 memory layout. Access
it through the **bounded little-endian accessors** on `api->game_ram`: `u8`/`u16`/`u32`,
`set_u8`/`set_u16`/`set_u32`, `read_block`/`write_block`, `size`, and `to_ptr`. Offsets are
canonical addresses (the frozen 1996 layout). All access is bounds-checked against `size()`:
out-of-range **reads return 0**, out-of-range **writes are dropped**, and block ops return **−1**.

`to_ptr(off)` resolves a canonical address to its host pointer (base + off; `NULL` if out of
range) — use it to form a `game_ram` pointer to pass to an exposed engine function, or to match a
runtime code/data pointer the engine stored (which it writes as address + base).

**Pointer/lifetime rule:** the engine's DAS pool allocator relocates its chunks, so **never cache
a pool-derived pointer across an engine call or a tick** — re-resolve through the handle each time.

## Execution contexts

Callbacks and overrides run in one of four platform-neutral contexts, a frozen enum:

- `ROTH_CTX_MAIN` — boot / main thread: discovery, validation, patching, all before any engine,
  timer, or audio thread starts. Anything is legal.
- `ROTH_CTX_GAME` — the game thread: per-frame logic. Engine calls are legal.
- `ROTH_CTX_TICK_ISR` — the compositor/timer tick: a restricted API view. **No engine calls**;
  keep it fast.
- `ROTH_CTX_AUDIO` — reserved; a restricted API view.

Every callback typedef is context-suffixed (`..._main_fn`, `..._game_fn`, `..._tick_isr_fn`), and
the enumerators are frozen ABI — never renumbered.

## The override-target manifest

`abi/override_targets.json` is the authority on which functions a mod may override. It lists
**1132 override-eligible** functions; `roth_sdk_overrides.h` emits a typed `ROTH_FN_<name>`
enumerator and chain machinery for each. **11 functions are hard-excluded** — the timer/ISR bodies
and audio callbacks, the engine's pulse — because overriding them would break timing rather than
gameplay. The manifest is reproducible from `functions.json` plus a fixed policy and carries no
addresses; the per-build address descriptor (which targets are present, and where) is produced at
build time. Each eligible entry also carries advisory flags: `tick_isr_reachable` (the override
may run in the 70 Hz timer tick, so keep it fast and self-contained) and `self_modifying` (the
original body contains self-modifying-code patterns; replace only if you understand that
machinery).

## Versioning + the bump policy

The SDK and ABI are at **0.5** (`ROTH_SDK_MAJOR/MINOR`, `ROTH_ABI_MAJOR/MINOR` in `roth_sdk.h`;
`version` in `sdk_api.json`). The load rule the host enforces:

> a plugin loads **iff** `abi_major == host.abi_major` **and** `abi_minor <= host.abi_minor`.

Because the SDK only ever grows additively, a plugin built against an earlier 0.x SDK keeps loading
on every later 0.x host. The bump policy, enforced by CI against the frozen baselines in
`abi/released/`:

- **identical** ABI ⇒ versions must be equal;
- **append-only** change (a new trailing struct field, a new appended engine slot) ⇒ **minor** bump;
- **breaking** change (a removed/moved/resized field, a changed signature) ⇒ **major** bump.

## Regenerate + verify

```
make -C sdk gen      # regenerate the generated files from the schemas
make -C sdk check    # run all offline CI gates
```

`make -C sdk check` runs seven gates, all offline (no network, no writes to the repo):

1. **drift** — regenerate every generated file in memory and diff against what's committed; any
   difference fails (the schemas are the source, the generated files must be reproducible).
2. **id-invariants** — `function_ids.json`: unique IDs; every name (canonical or alias) resolves to
   exactly one ID; every corpus address present exactly once; `canonical_name` matches the current
   source symbol; and versus the released baseline, IDs are never removed/renumbered and aliases
   are never dropped.
3. **abi-diff** — classify the descriptor change versus `abi/released/` (identical / append-only /
   breaking) and enforce the bump policy.
4. **exposure-order** — THE EXPOSURE-ORDER LAW: the released baseline's engine-table slot sequence
   must be an exact prefix of the current one; any re-sort/insert/removal fails hard, no bump
   excuses it.
5. **compile** — `roth_sdk.h` syntax-only and `host_adapters.c -c` under
   `gcc -m32 -std=c11 -Wall -Wextra -Werror` (the ABI proof).
6. **abi-probe** — compile + run an `offsetof`/`sizeof` probe under `-m32` and check the real i386
   layout against `descriptor.json` (SKIPs, never fails, if a 32-bit build/run is unavailable).
7. **override-validate** — validate the override machinery (entry-pad presence, no branches landing
   inside an eligible entry region, dispatch-edge targets resolve to function entries) against the
   linked moddable binary (SKIPs if it isn't built).

## Not currently supported

Per-function source fingerprints and changed-function release manifests, zip-packaged mod
archives, a scripting (Lua) tier, the Windows port, and `on_audio` dispatch are not present. The
append-only versioning above is designed to accommodate each as ordinary additive growth.
